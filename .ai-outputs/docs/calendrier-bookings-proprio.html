📅 Agenda global proprio + résa manuelle

Avant cette feature, le proprio devait ouvrir chaque annonce une par une pour voir son calendrier, et il n'avait aucun moyen de saisir une réservation faite hors plateforme (client direct, paiement en espèces, démarcheur partenaire). Cette feature livre une vue agenda unique multi-annonces avec stats du mois, un bandeau Conseil chiffré qui pousse à optimiser le revenu, et un wizard 3 étapes pour créer une réservation manuelle traçable.

⚖️ Règles métier

Stats du mois

StatCalculTone
Occupéjours du mois avec une plage OCCUPE ou EN_ATTENTEdanger
LibrejoursDuMois − Occupé − Bloqué propriosuccess
Manque à gagner(Libre + Bloqué) × prixNuitaccent

Le « Manque à gagner » inclut volontairement les jours bloqués par le proprio. Objectif métier : inciter à débloquer ce qui peut l'être (maintenance prolongée, perso non urgent).

Conseil hebdomadaire (TipSuggestionEngine)

  1. Compter joursLibres dans la semaine en cours (lundi → dimanche, blocages proprio comptent comme libres)
  2. Si joursLibres >= 4 → afficher le bandeau ; sinon masquer
  3. Suggestion = min(joursLibres, 4) × prixNuit × tauxOccupationMoyen (fallback 70 %)

Wizard de création — source enum

Valeurvalue (JSON)Commission
clientDirectCLIENT_DIRECT0 % (proprio reçoit tout)
demarcheurPartenaireDEMARCHEUR_PARTENAIRE10 % au démarcheur

« Via Asfar » est volontairement absent du wizard : une réservation manuelle est par définition hors plateforme.

Mode de paiement (tracking proprio)

4 chips proposés au wizard step 2 (extraits de MoyenPaiement.manualReservationOptions) : Espèces, Wave, Orange Money, Virement. Le moyen de paiement est purement informatif côté proprio — il n'affecte aucun calcul de commission Asfar.

Validation publication

  • Dates : pas de chevauchement avec plage occupée ou en attente. Dates passées autorisées (résa rétroactive).
  • Client : nom + téléphone obligatoires (non vides après trim).
  • Source : obligatoire ; si demarcheurPartenaire → demarcheurId requis.
  • Moyen de paiement : obligatoire (1 chip sélectionné).

🎬 Flow utilisateur — Création réservation manuelle

  1. Proprio ouvre le dashboard → tape sur la card « Calendrier & bookings »
  2. Écran s'ouvre sur la 1re annonce du proprio. Chips horizontales sticky permettent de switcher
  3. Tape sur le bouton + Bloquer / Réserver en AppBar
  4. Bottom sheet 2 options. Choisit « Réserver pour un client direct »
  5. Step 1 : sélectionne 16-17 nov sur le calendrier date range (jours rouges = déjà occupés non-tappables)
  6. Step 2 : saisit nom (« Rachid B. »), téléphone, choisit Client direct, choisit Wave. Récap : 1 nuit × 68k = 68 000, « Vous recevez 68 000 »
  7. Step 3 : confirmation « Réservation enregistrée — réf ASF-H8R2X » avec SuccessCircle vert et récap final
  8. Tape « Retour au calendrier » → 16-17 nov est désormais marqué occupé, la stat « Réservations du mois » s'incrémente

🔌 Coordination backend

L'endpoint createManualReservation existe déjà côté serveur. Le wizard envoie 3 champs supplémentaires sur le payload :

{
  "appartId": 12,
  "debut": "2026-11-16T00:00:00.000",
  "dure": 1,
  "clientNom": "Madame Touré",
  "clientTelephone": "+225 07 12 34 56",
  "montant": 68000,
  "source": "CLIENT_DIRECT",          // ← NOUVEAU
  "moyenPaiement": "WAVE",            // ← NOUVEAU
  "demarcheurId": null                // ← NOUVEAU (requis si source = DEMARCHEUR_PARTENAIRE)
}

⚠️ Demande backend

Pour l'instant, ces 3 nouveaux champs sont envoyés mais ignorables côté serveur (parser tolérant). Côté Flutter, leur absence dans le DTO retourné n'a pas d'impact. Pour activer la commission démarcheur sur résa manuelle et la traçabilité paiement, le backend doit :

  • Persister source et moyenPaiement sur ReservationManuelle (2 colonnes enum)
  • Si source == DEMARCHEUR_PARTENAIRE : persister demarcheur_id (FK), calculer montantCommission = montant × 0.10

Détails complets dans BACKEND_NOTES_RESERVATION_DETAIL.md §0.
