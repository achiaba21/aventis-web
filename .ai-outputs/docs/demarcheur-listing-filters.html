📋 Description

Cette feature couvre deux améliorations de l'écran DemarcheurListingsScreen (flow "Nouvelle demande" — sélection d'un logement partenaire) :

A — Calendrier de disponibilités inline

Quand le démarcheur tape sur une card de logement, cette dernière se sélectionne (indicateur radio → coche) et un mini-calendrier apparaît en dessous, montrant les jours réservés et libres du mois courant. La navigation mois précédent/suivant est disponible, avec un plancher au mois en cours. Réutilise MiniCalendarGrid sans duplication.

B — Système de filtres

Un bouton "Filtrer" dans l'AppBar ouvre l'écran ListingFilterScreen avec 3 sections dynamiques : Pièces (chips multi-select), Partenaire (bottom-sheet picker), Zone (bottom-sheet picker). Les filtres se combinent en logique AND et un badge numérique sur le bouton indique le nombre de filtres actifs.

C — Infrastructure vue carte (R14 — backend requis)

Le widget ListingMapView (FlutterMap) est câblé et conditionné par le flag _showMap = false. L'activation ne requiert qu'un changement de valeur côté mobile une fois que le backend expose lat/lon dans le DTO.

📐 Règles métier

Filtres

R1

Bouton "Filtrer" dans l'AppBar → ouvre ListingFilterScreen (page complète, pas bottom-sheet).

R2

Filtres combinables en AND : seuls les logements vérifiant TOUS les filtres actifs s'affichent.

R3

Pièces : chips AppartementTypeLocation présents dans le dataset — sélection multiple (OR intra-section).

R4

Partenaire : liste des proprietaire uniques — sélection unique.

R5

Zone : liste des communeNom uniques — sélection unique.

R6

Badge numérique sur "Filtrer" si au moins 1 filtre actif (1 par section, max 3).

R7

Bouton "Réinitialiser" dans l'AppBar de ListingFilterScreen visible uniquement si !_draft.isEmpty.

R8

0 résultat après filtres → EmptyState.inline + CTA "Réinitialiser les filtres" (pas EmptyState.hero).

R9 — Sections dynamiques

Une section est masquée si le dataset ne contient aucune valeur distincte pour ce filtre.

Calendrier inline

RC1

Tap sur un logement → sélection radio + chargement du calendrier via CalendarService().getDemarcheurCalendar(id).

RC2

Résultats mis en cache : _calendarCache[id] — un seul appel réseau par logement par session.

RC3

Borne min : mois courant — la navigation "mois précédent" est désactivée si on est déjà sur le mois courant.

Vue carte (R14)

R13

Vue carte masquée (_showMap = false) jusqu'à ce que le backend expose lat/lon.

R14

Le backend doit ajouter lat et lon obfusqués dans AppartementForDemarcheurDto — GET /api/demarcheur/appartements.

R15

Coordonnées obfusquées (comme MapAppartement), pas les coordonnées réelles.

Flux de données — Filtres

1
DemarcheurBloc → DemarcheurDataLoaded

Les appartements sont déjà en mémoire (chargés au initState).

2
Tap "Filtrer" → _openFilters()

Lit le BLoC à l'instant du tap pour extraire allApparts. pushScreen<ListingFilters> vers ListingFilterScreen.

3
ListingFilterScreen → retour ListingFilters

L'utilisateur ajuste les filtres, tape "Appliquer" → Navigator.pop(_draft).

4
setState → _activeFilters = result

_activeFilters.apply(allApparts) produit la liste filtrée. Badge mis à jour via _activeFilters.activeCount.

Flux de données — Calendrier inline

1
Tap sur PartnerListingCard → _selectListing(appart)

Guard if (id == null) return. setState _selectedId = id.

2
Cache miss → CalendarService().getDemarcheurCalendar(id)

Appel réseau uniquement si !_calendarCache.containsKey(id). Ajoute id à _loadingIds pendant le chargement.

3
Résultat → _calendarCache[id] = response

setState retire l'id de _loadingIds. La card sélectionnée reçoit calendarWidget: ListingAvailabilityCalendar(data: _calendarCache[id]).

Pattern d'obfuscation recommandé (backend)

Même approche que MapAppartement pour les logements du module locataire : appliquer un offset pseudo-aléatoire déterministe par appart_id (±0.003° à ±0.007°) pour préserver la cohérence carte-à-carte tout en masquant l'adresse exacte.
