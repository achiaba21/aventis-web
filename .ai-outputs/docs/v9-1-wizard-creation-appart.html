Vue d'ensemble

V9.1 introduit l'écran ProprioNewListingScreen accessible depuis ProprioListingsScreen via le bouton NewListingCard en bas de la liste des annonces. L'écran orchestre :

  • Un BlocProvider<AppartementWizardBloc> local au screen — le draft Hive n'est pas partagé entre instances
  • Un BlocListener qui réagit à 3 signaux du state : hasResumableDraft (modal reprise), published (déclenche upload async), validationErrors (SnackBar)
  • 5 sous-widgets d'étape switching selon state.currentStep
  • Auto-save debouncé (Timer 500ms) qui dispatch TriggerAutoSave après chaque UpdateField
  • Capture GPS via GeoLocationService (réutilise le service existant) avec preview MiniMapPreview V9.7c
  • image_picker.pickMultiImage(limit: 8) pour les photos, conversion en UploadedImage au publish
  • Publication finale via AppartementRepository().createAppartementWithImages (couplage faible Bloc/Service documenté dans le Bloc)

Parcours utilisateur

Flow nominal — création annonce

  1. Proprio ouvre ProprioListingsScreen (onglet Annonces de son shell)
  2. Tap sur la NewListingCard en bas de la liste → pushScreen(ProprioNewListingScreen)
  3. BlocProvider instancie AppartementWizardBloc, dispatch InitWizard()
  4. Si hasResumableDraft == true → ResumeDraftDialog apparait → user choisit Reprendre ou Recommencer
  5. Sinon → FetchInitialLocation dispatché automatiquement (préremplit l'adresse depuis la position téléphone)
  6. Étape 1 → user sélectionne une RoomsTypeCard → UpdateField('typeLocation', value) → Continuer
  7. Étape 2 → titre/ville/commune/quartier saisis, GPS capturé via bouton "Activer GPS" → coords stockées dans draft.address.lat/longi → Continuer
  8. Étape 3 → tap PhotosUploadCard → image_picker.pickMultiImage(limit: 8) → photos converties en PhotoAppart(path: xfile.path) → grille avec badge "Couverture" sur la 1ère → Continuer (gating min 3)
  9. Étape 4 → toggle équipements → List<Offre>(Commodite(nom: label)) → Continuer
  10. Étape 5 → saisit prix → preview commission auto → toggles règles → Publier mon annonce
  11. Bloc dispatch PublishAppartement → validator → si OK state.published = true
  12. BlocListener déclenche _publish() async → List<UploadedImage> depuis paths → AppartementRepository().createAppartementWithImages(draft, images)
  13. Sur succès : DiscardDraft + AppartementBloc.RefreshAppartements + back(context) + SnackBar "Annonce publiée"

Flow auto-save brouillon

  1. User modifie un champ → UpdateField(field, value) dispatché au Bloc
  2. Bloc met à jour state.draft + recalcule canPublish via validator
  3. Screen Timer 500ms debounce démarré (chaque modif reset le timer)
  4. Au timeout → dispatch TriggerAutoSave → Bloc → AppartementDraftStorage.save(draft) en Hive
  5. Si user quitte app/écran → draft persisté → reprise à la prochaine ouverture du tunnel

Flow géolocalisation

  1. Tap "Activer GPS" dans GpsCaptureCard → dispatch FetchInitialLocation
  2. Bloc → GeoLocationService.requestPermission()
  3. Si refusée → state.isLoadingGeo = false sans coords (acceptable, backend fera fallback géocoding)
  4. Si accordée → GeoLocationService.getCurrentLocation() + reverseGeocode
  5. Coords stockées dans draft.address.lat/longi + nom adresse reverse → preview MiniMapPreview V9.7c affichée en dessous
  6. User peut "Recapturer" à tout moment

Flow erreur publication

  1. Tap "Publier mon annonce" → Bloc emit published = true, isPublishing = true
  2. BlocListener déclenche _publish async
  3. Erreur réseau / 400 / 401 → catch dans _publish → SnackBar "Échec de la publication: ..." → _publishStarted = false (retry possible)
  4. Draft Hive intact (purge uniquement après succès 200) → retry sans perte de données

Décisions techniques actées

1. Réutilisation 100% infrastructure existante

L'audit code a révélé que AppartementWizardBloc + AppartementDraftStorage + GeoLocationService + AppartementPublicationValidator + AppartementBackendMapper + AppartementRepository existent déjà. Aucune duplication. V9.1 ne crée que des widgets UI qui dispatchent UpdateField/NextStep/PublishAppartement/... au Bloc existant.

2. Hive draft auto-save Timer 500ms debounce

Pendant la saisie, l'écran maintient un Timer? dispose-safe :

void _scheduleAutoSave() {
  _autoSaveDebouncer?.cancel();
  _autoSaveDebouncer = Timer(const Duration(milliseconds: 500), () {
    if (!mounted) return;
    context.read<AppartementWizardBloc>().add(TriggerAutoSave());
  });
}

Cohérent : pas de spinner pendant la frappe, save invisible une fois la frappe stabilisée. Le Bloc gère le _draftStorage.save(draft) Hive sync.

3. Cross-feature reuse — MiniMapPreview V9.7c dans GpsCaptureCard

Le MiniMapPreview créé en V9.7c (mini-carte position post-réservation) est réutilisé tel quel dans le GpsCaptureCard de V9.1 pour afficher la position téléphone capturée. Démontre la valeur du factor commun extrait en V9.7c (osmDarkMatrix partagé).

4. Cross-feature reuse — DashedBorderContainer V7 dans PhotosUploadCard

Le DashedBorderContainer (atome CustomPainter créé en V7 pour la NewListingCard) est réutilisé pour la card "Téléverser depuis l'appareil" du wizard. Cohérence visuelle Asfar sur les CTA dashed (pattern card "à ajouter").

5. AsfarToggle custom proto-fidèle

Au lieu d'utiliser SwitchListTile Material/Cupertino, V9.1 introduit un AsfarToggle custom — AnimatedContainer 200ms ease-out avec bullet blanc 22×22 et fond accent/bgElev3. Fidèle exact au proto proprietaire-extras.jsx::Toggle lignes 474-488 et identitaire Asfar. Réutilisable autres écrans (Profile settings, etc.).

6. Couplage faible Bloc / Service de publication

Le AppartementWizardBloc est explicitement décrit comme "NE FAIT PAS d'appel API" (cf. docstring). Quand la validation passe, il emit juste published = true. Le screen écoute ce flag via BlocListener et déclenche lui-même l'appel AppartementRepository().createAppartementWithImages. Avantage : le Bloc reste indépendant de l'infrastructure réseau, testable unitairement sans mock service.

7. Règles toggles sérialisées en string opaque

Les 3 toggles step 5 (Démarcheurs · Caution · Animaux) n'ont pas de champs dédiés dans le modèle Appartement. Solution MVP : sérialisation simple dans le champ regles existant :

final rulesJson = 'demarcheurs=true;caution=true;animaux=false';
final updated = state.draft.copyWith(regles: rulesJson);

TODO V10 : aligner backend pour exposer des champs typés (acceptsAgents: bool, requiresDeposit: bool, petsAllowed: bool).

8. Frais de ménage = visuel-only MVP

Le proto inclut un input "Frais de ménage (optionnel)" en étape 5. Le modèle Appartement côté Flutter n'a pas de champ dédié. V9.1 affiche un _CleaningFeePlaceholder visuel-only (fidélité proto) mais ne persiste pas la valeur. TODO V10 : ajouter cleaningFee: int? au modèle Appartement quand le backend l'expose.
