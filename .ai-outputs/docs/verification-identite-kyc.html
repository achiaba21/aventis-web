🛡️ Vérification d'identité (KYC) Audit 98/100

Envoi et suivi des pièces justificatives (propriétaire / démarcheur), statut « vérifié » et rafraîchissement temps réel au verdict admin

Vue d'ensemble

Un propriétaire ou un démarcheur envoie une photo de pièce d'identité (titre choisi dans une liste). Un admin l'approuve ou la rejette. L'utilisateur est « vérifié » dès qu'au moins un document est au statut VERIFIER. L'écran se rafraîchit automatiquement au verdict admin via le canal de notifications existant. Le locataire n'a pas accès à la fonctionnalité.

Statuts d'un document

Statut backendAffichageTon
EN_ATTENTEEn attente⏳ warning
VERIFIERVérifié✅ success
REFUSERRefusé (+ motif)❌ danger

Statut global dérivé (pas d'endpoint dédié) : verified si ≥ 1 document vérifié, sinon pending si au moins un document, sinon none.

Parcours utilisateur

  • Profil → entrée « Vérification d'identité » (affiche le vrai statut : Vérifié / En attente / Non vérifié).
  • Écran KYC : bandeau de statut global + historique complet des documents (miniature, titre, date, badge, motif si refusé).
  • Bouton « Envoyer une pièce » (ou « Renvoyer une pièce » après un refus) → bottom sheet : titre (liste prédéfinie ou « Autre ») + photo (galerie/caméra) + aperçu → envoi.
  • Après upload : document en EN_ATTENTE. À l'approbation/rejet, notification reçue → l'écran se recharge automatiquement.

API consommée

MéthodeEndpointRôle
GET/api/user/documentsListe des documents (tous statuts)
POST/api/user/documentsUpload multipart : file + titre

Affichage du fichier : ${domain}/${path} (servi en statique, sans auth). Le token Bearer est injecté automatiquement par DioRequest. Les erreurs 400 (fichier vide, > 5 Mo, type non supporté, titre manquant, rôle non autorisé) sont relayées telles quelles à l'utilisateur (SnackBar).

Règles métier

  • Seuls Propriétaire et Démarcheur peuvent envoyer.
  • Envois illimités ; chaque envoi = nouveau document (uuid distinct).
  • Après un rejet : pas de réédition — on renvoie un nouveau document ; l'ancien reste REFUSER (historique conservé).
  • Le serveur bloque certaines actions sensibles (ex. demande de partenariat) tant qu'aucun document n'est vérifié.

Périmètre v1 / hors périmètre

v1 : photo (galerie/caméra), titre prédéfini, liste + statuts + motif, détection « vérifié », renvoi après refus, refresh sur notif.

Hors v1 : upload PDF, endpoints admin, ré-édition d'un document refusé, cache local des documents.
