Version 1.0 · Créée le 2026-06-11 · Tunnel d'inscription en 5 écrans, sans email, PIN 5 chiffres au clavier dédié

🧭 Vue d'ensemble

L'inscription monolithique (formulaire 6 champs — nom, prénom, téléphone, email, mot de passe, confirmation — puis OTP en fin de parcours) est remplacée par un tunnel court où le numéro est vérifié avant toute autre saisie. L'email et le prénom ne sont plus demandés (complétés plus tard dans le profil). Le flux est identique pour les 3 rôles (locataire, propriétaire, démarcheur), le rôle restant sélectionné en amont via l'onboarding. 

1. Numéro (ÉTAPE 1/4)  →  2. OTP 4 chiffres (2/4)  →  3. Nom complet (3/4)
                       →  4. Code secret PIN 5 chiffres (4/4)  →  5. Confirmation (4/4)  →  shell du rôle

Règle structurante UI (option B1 validée) : une page = une saisie = un clavier. Les pages PIN utilisent un clavier numérique dédié (PinKeypad), jamais le clavier système.

📐 Règles métier

  • Numéro déjà utilisé : refusé par le backend dès POST /auth/otp/send — l'erreur s'affiche à l'étape 1, aucun SMS envoyé.
  • OTP : code à 4 chiffres, soumission automatique à la complétion. Tentatives et blocage (5 max) gérés côté backend.
  • Renvoi progressif : délais 15s → 20s → 30s → 60s (plafonné), lien grisé pendant le cooldown — OtpResendPolicy.delayFor(resendCount).
  • PIN : exactement 5 chiffres, confirmé sur une page séparée ; la concordance est vérifiée localement (aucun appel serveur) — mismatch = cellules en danger + saisie réinitialisée.
  • Pas de retour sur un OTP consommé : l'écran OTP est remplacé (pushScreenAndReplace) par l'étape nom après vérification.
  • Création du compte à la confirmation du PIN : UserReq{nom, telephone, password=pin, confirmePass=pin, type=rôle} — email/prénom null.

🔄 Changements BLoC

L'ancien événement VerifyAndSignup(code, userReq) (vérification + création en un coup) est supprimé : le nouveau flux exige de vérifier l'OTP avant de connaître le mot de passe.

ÉlémentRôle
SendOtp(telephone) → OtpSentInchangé — étape 1 et renvois
VerifyOtp(telephone, code) → OtpVerified NOUVEAUVérification seule, déclenche l'étape 3
SignupUser(userReq) → UserLoadedExistant, réutilisé à la confirmation du PIN

UserBloc accepte désormais une injection optionnelle (UserBloc(authentication: …)) pour les tests.
