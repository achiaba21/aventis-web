Refonte enregistrement appartement

Suppression de la notion de Résidence + wizard d'ajout style Airbnb, auto-save silencieux, géolocalisation auto-détectée, et migration transparente des données existantes.

Livrée v1.0 Audit 92.5/100

1. Contexte

L'application Asfar permettait au propriétaire d'enregistrer ses biens locatifs en deux temps : d'abord créer une Résidence (nom + adresse), puis y rattacher un ou plusieurs Appartements.

Cette structure dupliquait des écrans, complexifiait la maintenance, et imposait au proprio (souvent avec 1 à 5 biens à des adresses distinctes) de comprendre une notion intermédiaire sans valeur perçue.

Demande utilisateur : abandonner totalement la notion de Résidence, intégrer l'adresse directement dans l'Appartement, et refondre l'UI d'ajout pour qu'elle soit « simple, sans prise de tête, fluide et agréable ».

2. Objectifs

  • Supprimer complètement la classe Residence du code applicatif (modèle, BLoC, repos, services, écrans, widgets).
  • Faire de Address un attribut direct de Appartement.
  • Refondre l'UI de création/édition en wizard 5 étapes guidées, avec auto-save silencieux du brouillon.
  • Préserver la compatibilité avec le backend Spring Boot non encore migré, via une couche d'adaptation isolée.
  • Migrer automatiquement les données existantes (cache local Hive) au premier lancement de la nouvelle version.

3. Parcours utilisateur (wizard 5 étapes)

Style validé : Option C "Hybrid Smart" — densité adaptative selon le type de contenu de chaque étape.

Adresse

Carte plein écran + bottom sheet avec adresse texte. GPS auto-détecté.

Décrire

Titre (obligatoire) + sélecteur visuel du type de location.

Capacité

Pills 1/2/3/4+ pour chambres, steppers pour lits/douches.

Photos & équipements

Min 3 photos pour publier + chips équipements.

Récap

Aperçu Airbnb-like + prix + bouton « Publier ✨ ».

Comportements clés

  • Auto-save silencieux à chaque étape franchie + debounce 500ms sur les saisies.
  • Reprise de brouillon : modal au lancement si un draft existe.
  • Validation par étape : titre, adresse, capacité, prix, ≥3 photos.
  • Mode édition : même wizard, pré-rempli, ouverture optionnelle sur une étape via deep-link.

6. Migration & compatibilité backend

Migration locale Hive (au démarrage)

LegacyResidenceMigration.instance.runIfNeeded() est appelé dans main() juste après StorageService.instance.init(). Algorithme :

  1. Vérifier le flag migration_v2_appart_address_done dans la box app_settings. Si présent → noop.
  2. Lire les résidences en JSON brut (Map) depuis Hive.
  3. Pour chaque appartement, fusionner residence.address → appart.address (si non déjà renseigné).
  4. Sauvegarder les apparts migrés, vider la box résidences.
  5. Poser le flag.
Idempotente : rejouer la migration plusieurs fois ne casse rien — le flag empêche toute exécution ultérieure. Les appartements orphelins (sans résidence parente) sont conservés avec address = null, à compléter manuellement par le proprio.

Compatibilité backend Spring Boot

Le backend n'est pas encore migré au modèle plat. Le mapper AppartementBackendMapper isole cette dette : 

// À l'envoi (création) :
final payload = appart.toJson();
payload.remove('address');
payload['residence'] = _buildLegacyResidenceShape(appart);
// Le backend reçoit { residence: { address: {...} } } comme avant.

// À la réception :
final addressMap = _extractAddressFromLegacy(json);
json.remove('residence');
json.remove('residenceId');
final appart = Appartement.fromJson(json);
appart.address ??= Address.fromJson(addressMap);
TODO BACKEND-FLAT-APPART : quand le backend Spring acceptera appartement.address directement, supprimer AppartementBackendMapper et la fusion défensive dans Appartement.fromJson. Le reste du code applicatif n'aura aucune adaptation à faire.
