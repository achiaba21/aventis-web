🔒 Sécurisation de l'application mobile Validé 95/100

Version 1.0 · Créée le 2026-06-10 · Implémente les fiches d'audit SEC-01 à SEC-05 (.ai-outputs/audit/) · Specs : .ai-outputs/specs/securite-mobile/

📋 Description

Mise au niveau « production » de la sécurité de l'app : transport chiffré par défaut, jeton de session dans le Keychain/Keystore, cache Hive chiffré, clé cartographique sortie du code source, journaux sans données sensibles, sessions qui expirent proprement. Aucun changement visible pour l'utilisateur final, hormis une reconnexion unique à la première ouverture (pas de migration des données : aucun utilisateur réel en production).

🚀 Commandes de build

Développement (serveur local sans TLS)

flutter run --dart-define=USE_TLS=false \
            --dart-define=SERVER_HOST=192.168.1.7 \
            --dart-define=SERVER_PORT=7565

Sans USE_TLS=false, l'app force https:// et wss:// (défaut sûr). Côté Android, le trafic en clair n'est autorisé que par la variante debug de network_security_config.xml — un build release le refuse toujours.

Production

flutter build apk \
  --dart-define=SERVER_HOST=api.mondomaine.com \
  --dart-define=SERVER_PORT=443 \
  --dart-define=STADIA_KEY=<nouvelle-clé>   # optionnel tant que les cartes sont sur OSM

🧩 Nouveaux composants

lib/service/storage/secure_storage_service.dart

  • init() — pré-charge le jeton (à appeler avant StorageService.init())
  • cachedToken — lecture synchrone du jeton
  • saveToken / deleteToken — écriture Keychain/Keystore
  • getOrCreateHiveKey() — clé AES 256 des boxes Hive

lib/service/auth/token_validator.dart

  • TokenValidator.isValid(String?) — présent + décodable + non expiré, ne lève jamais d'exception
