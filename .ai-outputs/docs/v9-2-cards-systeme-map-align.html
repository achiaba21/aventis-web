Parcours utilisateur

Flow 1 — Locataire reçoit une card Réservation après paiement

  1. Locataire paie sa résa via POST /api/user/reservations/{ref}/pay
  2. Backend émet un ChatMessage avec isSystem: true, contenu [ASFAR_CARD:reservation]{"ref":"ASF-7K2N9"}
  3. Au prochain GET messages, Flutter reçoit le ChatMessage avec isSystem: true
  4. ChatMessageToUiMapper.detectKind retourne reservationCard (isSystem ✓ + préfixe ✓)
  5. _parseReservation extrait {ref: "ASF-7K2N9"} → ReservationCardPayload(reference: "ASF-7K2N9")
  6. ThreadMessageItem rend ReservationMessageCard qui démarre initState → skeleton 3 zones gris
  7. ReservationService.getByReference("ASF-7K2N9") → GET /api/user/reservations/ASF-7K2N9
  8. Backend retourne Reservation complète (avec appart, dates, prix)
  9. setState → skeleton swap pour titre appart + dates + code mono accent or
  10. Locataire tap → onTap(loaded) → MessagingThreadScreen._onReservationTap push LocataireDetailScreen (focus appart via AppartementToListingMapper.mapOne)

Flow 2 — Démarcheur reçoit une card Partenariat acceptée

  1. Proprio accepte via POST /api/proprietaire/partenariat/demandes/{id}/accepter
  2. Backend émet ChatMessage isSystem: true contenu [ASFAR_CARD:partenariat]{"id":12}
  3. Mapper détecte → acceptedPartenariatCard, parse {id: 12} → AcceptedPartenariatCardPayload(demandeId: 12)
  4. AcceptedPartenariatMessageCard.initState → skeleton 2 zones + appel PartenariatService.getDemandeById(12)
  5. GET /api/demande-partenariat/12 → DemandePartenariat complet
  6. Card calcule le nom à afficher = **partie opposée** (si user démarcheur → nom proprio, si user proprio → nom démarcheur) via UserBloc.state.user.type
  7. swap content : nom partie + "Accepté le 11 mai"
  8. Tap → push PartenariatDetailScreen(demande: loaded) — écran transverse
  9. PartenariatDetailScreen affiche : statut large chip success/warn/danger + dates + 2 cards parties (démarcheur + proprio avec avatar gradient or + bouton phone tel:)

Flow erreur fetch lazy

  1. Card mount → tentative fetch
  2. Si fetch échoue (réseau, 403, 404) → _failed = true
  3. Card affiche : titre fallback ("Réservation ASF-XXX" ou "Partenariat #12") + chip "Indisponible" text3 muted (couleur silencieuse, pas warn/danger)
  4. Tap card avec loaded = null → SnackBar "Détail indisponible" (pas de push)
  5. Comportement coherent UX V9.7c (fallback silencieux, expérience luxe)

Décisions techniques actées

1. Hive isSystem sans bump typeId

Le modèle ChatMessage a @HiveTypeId(1) dans des boxes existantes en production. V9.2 ajoute @HiveField(9) bool? isSystem en nullable, ce qui permet à Hive de tolérer les boxes pré-V9.2 (champ absent → null → traité comme false). Le fichier chat_message.g.dart est mis à jour manuellement pour persister le nouveau champ (numFields 9→10, readByte/writeByte 9 ajoutés).

2. Atomes partagés system_card_atoms.dart

Les 2 cards système partagent leur structure visuelle (icon leading + skeleton + chip indisponible). Extraction de 3 atomes communs :

SystemCardLeadingIcon(icon: ...)        // 40×40 accentSoft + icon accent
SystemCardSkeletonRows(rowWidths: [...]) // zones bgElev2 statiques
SystemCardUnavailableChip()              // pill text3 muted + info_outline

Zéro duplication entre ReservationMessageCard et AcceptedPartenariatMessageCard.

3. Callback signature avec objet loaded

Avant V9.2 : onTap: VoidCallback? → parent ne savait pas si le fetch avait réussi. Après V9.2 : onTap: void Function(Reservation? loaded)? (et idem DemandePartenariat?). Le parent décide quoi pousser selon le résultat :

void _onReservationTap(Reservation? loaded) {
  if (loaded?.appart == null) {
    _toast('Détail indisponible');
    return;
  }
  pushScreen(context, LocataireDetailScreen(
    listing: AppartementToListingMapper.mapOne(loaded!.appart!),
  ));
}

4. Parsing JSON résilient

Le mapper wrappe chaque jsonDecode dans un try/catch. Si le JSON est malformé ou inattendu :

static ReservationCardPayload? _parseReservation(String contenu) {
  try {
    final jsonStr = contenu.substring(_reservationPrefix.length);
    final map = jsonDecode(jsonStr) as Map<String, dynamic>;
    final ref = map['ref'] as String?;
    if (ref == null || ref.isEmpty) return null;
    return ReservationCardPayload(reference: ref);
  } catch (e) {
    deboger('parseReservation: $e');
    return null;
  }
}

Si retour null côté mapper → fallback MessageKind.text, le contenu brut s'affiche en bubble texte. Aucun crash possible sur message mal formé.

5. Conversation mixte proprio↔démarcheur

Le backend supporte désormais des seances entre Proprio et Démarcheur (pas que Proprio↔Locataire). ConversationToPreviewMapper._roleFor élargi :

if (myType == 'proprietaire') {
  if (otherType == 'demarcheur') return ConversationRole.demarcheur;
  return ConversationRole.tenant;
}

Le badge rôle dans la liste de conversations s'adapte automatiquement.

6. Cleanup AddressReq côté création appart

Le brief backend confirme que geoLat/geoLongi sont calculés automatiquement par geocoding serveur lors de la création d'une adresse. Côté Flutter, AppartementBackendMapper._buildLegacyResidenceShape les strippe désormais du payload address avant envoi :

final addressMap = appart.address!.toJson();
addressMap.remove('geoLat');
addressMap.remove('geoLongi');
shape['address'] = addressMap;

L'utilisateur continue de capturer lat/longi via GpsCapture (téléphone) — le backend prend le relais pour le geocoding.

7. PartenariatService neutre vs PartenariatProprioService

Le service existant PartenariatProprioService (V9.6) porte des actions liées au rôle proprio (accepterDemande, refuserDemande, getDemandes). Pour le lookup par ID utilisé par la card chat (besoin neutre, applicable proprio + démarcheur), V9.2 ajoute un nouveau PartenariatService singleton avec uniquement getDemandeById (route api/demande-partenariat/{id}). Séparation responsabilités propre.

8. Suppression de referral côté chat (renommage cascade)

Le terme referral côté Flutter chat est entièrement remplacé par partenariat pour aligner sur le backend :

  • Préfixe : [ASFAR_CARD:partenariat]
  • Payload : AcceptedPartenariatCardPayload (champ demandeId: int)
  • Widget : AcceptedPartenariatMessageCard
  • Endpoint : /api/demande-partenariat/{id}
  • Enum : MessageKind.acceptedPartenariatCard
  • Callback : onPartenariatTap

L'écran démarcheur ReferralDetailScreen V9.6 (autre flow : liste référence client côté démarcheur) reste inchangé.
