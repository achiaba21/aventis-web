🎯 Vue d'ensemble

La Vague 8 reconstruit la messagerie transverse en suivant fidèlement le prototype HTML (extras.jsx::MessagingList + ::MessagingThread). Cette messagerie est adaptative au rôle : un même utilisateur voit des conversations différentes selon qu'il navigue en mode Locataire, Démarcheur ou Propriétaire.

Avant la V8, les 3 Shells (Locataire V5, Démarcheur V6, Propriétaire V7) avaient un _MessagesPlaceholder indiquant « Messagerie à venir (Vague 8) ». La V8 supprime ces 3 placeholders et branche directement MessagingListScreen, qui lit dynamiquement le rôle actif via UserBloc.

« C'est le même compte sous différentes interfaces, ça doit être dynamique » — décision utilisateur actée 2026-05-10

Notifications (F6) et Receipt PDF preview (F8) ont été reportés à V9 (hors-proto) — décision BA pour scope V8 maîtrisé.

🧭 Parcours utilisateur

Scénario nominal — démarcheur ouvre la messagerie

  1. Démarcheur tap onglet Messages dans le DemarcheurShell
  2. MessagingListScreen lit UserBloc.state.user?.type = 'demarcheur'
  3. Charge SampleConversations.forRole('demarcheur') = 3 conversations (Aminata K. certifiée, Rachid B., M. Konaté)
  4. Affiche la liste avec badges rôle (info pour Démarcheur, accent pour Hôte/Client) + cercle unread accent or pour les non-lus
  5. Démarcheur tape « Aminata » dans la search bar → filtre local case-insensitive → 1 résultat
  6. Tap sur Aminata K. → pushScreen(MessagingThreadScreen(conversation: D1))
  7. Header custom affiché avec back + avatar + nom + shield certifié + sub « Hôte certifié · Loft Plateau » + bouton phone
  8. Liste des 5 messages avec séparateur « Aujourd'hui » + dernier message = card AcceptedReferralCardPayload (REF-D8H3K, commission 13 500 FCFA)
  9. Démarcheur saisit « OK je préviens le client » dans l'input bar → bouton send activé (opacity 1)
  10. Tap send → setState ajoute le message me en bas de la liste + scroll auto via WidgetsBinding.addPostFrameCallback

Scénario multi-rôle — switch via Profile

  1. Locataire tap onglet Profil → ClientProfileScreen V6 → tap « Démarcheur » dans ProfileRoleSwitcher
  2. RoleHomeRouter.shellFor push DemarcheurShell avec pushAndRemoveAll
  3. L'utilisateur arrive sur Démarcheur Dashboard. Tap onglet Messages → la messagerie affiche les conversations du rôle Démarcheur (pas locataire) — adaptation automatique via BlocBuilder

🎯 Décisions clés

1. MessagingList adaptatif au rôle

Le screen lit UserBloc.state.user?.type via BlocBuilder et appelle SampleConversations.forRole(role). Fallback locataire si rôle inconnu (cohérence proto extras.jsx:98 = convosByRole[role] || convosByRole.locataire). 

BlocBuilder<UserBloc, UserState>(
  builder: (context, state) {
    final allConvos = SampleConversations.forRole(state.user?.type);
    final visible = _filtered(allConvos);
    // ...
  },
)

2. Header thread custom (pas DynamicAppBar)

Le proto utilise un Container border-bottom + Row inline (back + avatar + Column[nom + sub] + phone). On reproduit fidèlement plutôt que de passer par DynamicAppBar V2 (qui ne supporte pas le double texte vertical avec shield inline). Décision proto-aligned.

3. Threads dynamiques (3 riches + autres vides)

Décision utilisateur acté : « ça doit être dynamique ». Le proto hardcode 1 thread par rôle (incohérent UX). Notre solution :

  • 3 threads riches fidèles proto :
    • L1 Aminata K. : 5 messages dont 1 reservationCard (Loft Plateau, 12-15 nov, ASF-7K2N9)
    • P1 Rachid B. : 4 messages texte (À quelle heure puis-je arriver ?)
    • D1 Aminata K. : 5 messages dont 1 acceptedReferralCard (REF-D8H3K, commission 13 500 FCFA)
  • Autres conversations : thread vide → placeholder « Démarrez la conversation… » centré
  • Header dynamique : adapté à la ConversationPreview cliquée (nom + sub + certified)

4. Cards spéciales fidèles au proto

Les 2 cards spéciales du proto sont reproduites au pixel-near :

  • Card Réservation (ReservationMessageCard) : Container bgElev1 82% + Row[ImgPh 56×56 + eyebrow RÉSERVATION + titre + dates + bookingCode mono]
  • Card Demande acceptée (AcceptedReferralMessageCard) : Container accentSoft + border accent×0.25 + icon check + label « Demande acceptée » + commission mono accent or

5. BlurContainer wrappe l'input bar

Le proto utilise rgba(10,10,11,0.92) flat. On enrichit avec BlurContainer V1 pour cohérence Liquid Glass Asfar (V5-V7 BottomNav, BottomBar). Écart documenté.

6. Bouton send désactivé visuellement si champ vide

Listener TextEditingController recalcule _canSend. Si vide, opacity 0.4 + tap ignoré. Si non vide, opacity 1 + tap envoyer + clear + scroll auto.

7. Suppression des 3 _MessagesPlaceholder

Les 3 classes privées _MessagesPlaceholder dans locataire_shell.dart, demarcheur_shell.dart, proprio_shell.dart ont été complètement supprimées et remplacées par const MessagingListScreen(). Vérification : grep -rn '_MessagesPlaceholder' lib/ retourne 0 occurrence.
