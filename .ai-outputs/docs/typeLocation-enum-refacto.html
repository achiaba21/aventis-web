typeLocation enum strict + dérivation nbChambres

Élimination d'un bug conceptuel : un champ String? portait deux sémantiques contradictoires (typologie de pièces côté wizard, type d'offre Airbnb-like côté édition), et le nombre de chambres était saisi deux fois sans cohérence croisée. 🔁 Le bug conceptuel résolu

Avant cette refacto, Appartement.typeLocation était une chaîne libre recevant tantôt "Studio", tantôt "3 pièces", tantôt "Appartement entier", tantôt "Chambre privée", selon que le proprio passait par le wizard ou par l'écran d'édition. Aucun lien avec nbChambres qui était saisi en parallèle. Conséquence : un Studio + 5 chambres était publiable, et un proprio pouvait écraser involontairement "3 pièces" par "Appartement entier" en éditant son annonce.

✅ Enum strict 5 valeurs ✅ nbChambres dérivé du type ★ UI inchangée step 1 /feature full pipeline

🧨 Contexte du bug

Avant

  • typeLocation: String? en chaîne libre
  • Wizard step 1 : enregistrait "Studio", "2 pièces", etc.
  • Édition (ListingInfosTab._editType) : TextFieldEditDialog avec hint suggérant un autre vocabulaire (Appartement entier / Studio / Chambre privée)
  • Step 2 du wizard redemandait nbChambres en stepper 0–10 libre
  • Aucune validation croisée → publications incohérentes possibles

Après

  • typeLocation: AppartementTypeLocation? (enum strict 5 valeurs)
  • Wizard step 1 inchangé visuellement, binding enum
  • Édition : TypeLocationEditDialog picker enum (RadioGroup)
  • Step 2 : stepper Chambres masqué sauf pour 5+ pièces (dérivation auto)
  • Validator refuse toute paire (type, nbChambres) incohérente

🧬 Modèle métier final

Enum AppartementTypeLocation

Convention métier : Studio = pièce unique sans salon. À partir de 2 pièces : 1 salon + n chambres. Le « salon » n'est pas un champ stocké : sa présence est implicite via l'enum.

Helper TypeLocationChambresPolicy

Helper statique pur qui isole les deux règles métier critiques (SRP) :

// Détermine la valeur à appliquer pour nbChambres selon le type :
//   Studio / 2P / 3P / 4P  → derived strict (1, 1, 2, 3)
//   CINQ_PLUS              → préserve current si ≥ 4, sinon force à 4
TypeLocationChambresPolicy.resolveNbChambres(type, current)

// Indique si une paire (type, nbChambres) est conforme aux règles :
//   Studio / 2P / 3P / 4P  → nbChambres == derived
//   CINQ_PLUS              → nbChambres >= 4
//   nbChambres == null     → false toujours
TypeLocationChambresPolicy.isCoherent(type, nbChambres)

Utilisé par 3 consommateurs : AppartementWizardBloc._applyField, ListingInfosTab._editType, et AppartementPublicationValidator.

🎨 Comportement UI

Step 1 du wizard (inchangé visuellement)

La grille 2 colonnes avec 5 RoomsTypeCard est conservée à l'identique. Seul le binding change : la sélection produit désormais une valeur enum typée au lieu d'une string libre. Le hint a été réécrit pour expliciter la nouvelle sémantique : « Studio = pièce unique sans salon. À partir de 2 pièces : 1 salon + chambres. »

Step 2 du wizard (Row capacité conditionnelle)

Selon le type sélectionné au step 1 :

  • Studio / 2 pièces / 3 pièces / 4 pièces → stepper Chambres masqué. La Row devient [Lits] [SdB] en 2 colonnes. nbChambres est dérivé en arrière-plan par le BLoC.
  • 5+ pièces → stepper Chambres visible avec min=4, max=10. La Row reste à 3 colonnes [Lits] [Chambres] [SdB] comme avant.

Édition (nouveau picker)

Dans ListingInfosTab, taper sur la FieldRow « TYPE » ouvre désormais TypeLocationEditDialog au lieu du TextFieldEditDialog. Le dialog est aligné sur CapacityEditDialog :

  • Title « Type de logement » + subtitle court
  • RadioGroup avec 5 RadioListTile (label + description courte)
  • Bouton « Enregistrer » + bouton « Annuler »

À la sauvegarde, si nbChambres est ajusté automatiquement (par exemple lors d'un passage de 5+ pièces 6 chambres vers Studio), une SnackBar informe le proprio : « Type changé en Studio, nombre de chambres ajusté à 1. »

🔄 Migration des annonces existantes

Le champ étant historiquement en chaîne libre, deux stratégies se combinent pour récupérer les annonces existantes :

1. Compat descendante côté Flutter

AppartementTypeLocation.fromBackend(raw) tente d'abord un matching strict sur les 5 valeurs enum, puis retombe sur fromLegacy(raw, nbChambres) qui combine deux pistes :

  1. Matching string (insensible casse) : détecte "studio", "2 pièces", "2p", etc.
  2. Dérivation depuis nbChambres pour les valeurs custom ("Chambre privée", "Appartement entier", vide, null) :
    • nbChambres ≥ 4 → CINQ_PLUS
    • nbChambres = 3 → QUATRE_PIECES
    • nbChambres = 2 → TROIS_PIECES
    • Autre (null, 0, 1) → DEUX_PIECES (default safe)

2. Migration backend (SQL one-shot)

Le mapping détaillé est documenté dans BACKEND_NOTES_ANNONCE.md §0. Coordination requise — déploiement possible avant ou après le client grâce à la compat descendante.
