Aperçu du ListingAvailabilityCalendar — jours réservés en accent or, libres en gris neutre, shimmer pendant le chargement.

Contexte

Dans le flow "Nouvelle demande" démarcheur, l'écran DemarcheurListingsScreen permettait déjà de choisir un logement parmi ceux proposés en partenariat. Cependant, aucune information sur la disponibilité du logement n'était accessible à cet instant.

Le démarcheur devait soit mémoriser les créneaux libres, soit revenir en arrière ou consulter une autre source, ce qui brisait le flow et augmentait le risque d'envoyer un client sur un logement déjà occupé.

« Je dois pouvoir savoir si le logement est libre avant d'envoyer mon client — sinon je peux me retrouver à faire une promesse que je ne peux pas tenir. »

Solution implémentée

Interaction de sélection

Tap sur un logement dans la liste → sélection radio activée + chargement inline du calendrier de disponibilités directement sous la card. L'utilisateur reste dans l'écran de choix, sans navigation supplémentaire.

Navigation mensuelle

Chevrons précédent / suivant pour naviguer entre les mois. Borne minimale = mois courant (impossible de revenir dans le passé). Le titre du mois se met à jour dynamiquement.

Légende et affichage

  • Jours réservés : fond accent/or — couvrent toutes les plages CONFIRME et EN_ATTENTE du CalendarResponse
  • Jours libres : fond neutre discret
  • Légende intégrée sous le calendrier : pastilles "Réservé" et "Libre"
  • État shimmer pendant le chargement réseau (pattern existant Asfar)
  • Hint texte contextuel si aucune donnée disponible

Bouton Continuer sticky

Un bouton "Continuer" apparaît en bas de l'écran, fixe (sticky), dès qu'un logement est sélectionné. Il reste visible même lors du scroll de la liste. Implémenté via Column + Expanded, sans Stack ni Positioned.

Titre écran

Le titre de l'écran a été changé de son ancien libellé vers "Choisir un logement" — formulation active, cohérente avec l'intention du flow.

Parcours utilisateur

  1. Dashboard démarcheur → tap CTA "Envoyer un client" → DemarcheurListingsScreen
  2. Titre "Choisir un logement" · liste des logements partenaires
  3. Tap sur un logement → _RadioIndicator passe de cercle vide à coche accent · bordure accent 1.5px activée
  4. Chargement inline → shimmer animé sous la card · CalendarService.getCalendar(appartId, month) appelé
  5. Calendrier affiché → mois courant · jours réservés/libres · légende · navigation mois
  6. Navigation mois → chevron suivant chargement mois+1 · chevron précédent désactivé si mois courant (borne min)
  7. Bouton "Continuer" sticky visible en bas → tap → étape suivante du tunnel (infos client)

Cas : changement de sélection

Si le démarcheur tape sur un autre logement alors qu'un premier est déjà sélectionné : la sélection se déplace, le calendrier se recharge pour le nouveau logement. Le cache en mémoire évite un appel réseau si le logement avait déjà été consulté pour le même mois.

Cas : logement déjà consulté (cache hit)

Si le démarcheur re-sélectionne un logement déjà chargé pour le mois en cours, le calendrier s'affiche instantanément depuis le _calendarCache (Map en mémoire), sans indicateur de chargement.

Décisions architecturales

1. Pas de nouveau BLoC — état UI éphémère dans le State

L'état de sélection (_selectedId), le mois affiché (_calendarMonth), le cache (_calendarCache) et les indicateurs de chargement (_loadingIds) sont gérés directement dans le State de DemarcheurListingsScreen. Ce choix est cohérent avec le pattern existant du tunnel (état éphémère du flow "Nouvelle demande", non persisté).

Un BLoC dédié aurait été surdimensionné pour un état purement local à cet écran, sans besoin de persistance ni de partage entre plusieurs widgets.

2. CalendarService appelé directement

CalendarService.getCalendar() est appelé directement depuis le State, sans passer par un BLoC intermédiaire. Ce pattern est cohérent avec les autres écrans du projet qui appellent des services directement lorsque l'état n'a pas besoin d'être partagé.

3. Réutilisation totale de MiniCalendarGrid

Le widget MiniCalendarGrid existant est wrappé par ListingAvailabilityCalendar sans aucune duplication de logique de rendu calendaire. Le nouveau widget se charge uniquement de :

  • Convertir CalendarResponse.plages en bookedDays et pendingDays pour le mois cible
  • Afficher le shimmer pendant le chargement
  • Rendre la légende Réservé / Libre
  • Afficher le hint texte si aucune donnée

Zéro duplication de la grille calendaire — règle DRY strictement respectée.

4. Cache en mémoire simple (V1)

Un Map<int, CalendarResponse> indexé par appartId sert de cache mémoire pour le mois affiché. Ce cache est non invalidé en V1 (durée de vie = durée de vie de l'écran). La navigation vers un autre mois vide le calendrier affiché mais ne purge pas le cache des autres logements.

Décision actée en V1 : cache invalidation et navigation multi-mois par logement sont des améliorations documentées pour une vague ultérieure.

5. Layout sticky "Continuer" via Column + Expanded

Le bouton "Continuer" est rendu sticky en bas grâce à un Column avec Expanded wrappant la liste scrollable — évite tout recours à Stack ou Positioned qui causeraient des conflits avec le layout iOS 26 (bug Container + alignment documenté en mémoire projet).

Points mineurs non bloquants (3)

  • Cache invalidation V1 — le cache mémoire Map<int, CalendarResponse> n'est pas invalidé lors d'un refresh. Comportement attendu en V1, documenté pour vague ultérieure.
  • Log fallback — l'erreur réseau dans _selectListing() est silencieuse (catch sans log). Amélioration mineure pour le debugging.
  • Docs constructeurs — les constructeurs de ListingAvailabilityCalendar et _ContinueButton n'ont pas de commentaires doc. Impact lisibilité uniquement.
