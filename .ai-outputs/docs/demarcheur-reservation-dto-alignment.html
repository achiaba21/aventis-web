1. Contexte backend

Le backend Spring a livré un nouveau DTO ReservationDemarcheurDto qui enrichit la réponse de GET /demarcheur/reservations avec des objets imbriqués et des champs scalaires additionnels.

Forme du nouveau payload

Section Champs
proprio nested id, nom, prenom, telephone, email, imgUrl
appart nested aplati id, numero, titre, imgUrl, prix, status, communeNom, villeNom
locataire nullable id, nom, prenom, telephone, email, imgUrl
demarcheur id, nom, prenom, telephone, email, imgUrl
avanceReservation nullable structure existante
scalaires type, reference, debut, fin, prix, frais, statut, motif, moyenPaiement, montantCommission, clientExterneNom, clientExterneTelephone, clientExterneEmail

2. Découverte clé

80% du parsing était déjà aligné côté mobile. Le scan d'architecture a révélé que la base Reservation.fromJsonCommon, ReservationDemarcheur, l'enum ReservationType, le système de contact (ContactTargetResolver + ContactSheet) et le ReferralDetailScreen étaient déjà câblés bout-en-bout sur le nouveau payload.

Ce qui était déjà en place

  • Reservation (abstract) parse proprio, locataire, appart, avanceReservation, type, clientExterne*, statut, motif, moyenPaiement, reference, prix, frais, debut, fin.
  • Factory polymorphique Reservation.fromJson qui crée la bonne sous-classe selon le champ type.
  • ReservationDemarcheur parse demarcheur et montantCommission.
  • Enum ReservationType complet (PLATEFORME / MANUELLE / DEMARCHEUR).
  • Système de contact : ContactTargetResolver.fromReservation(r, demarcheur) + ContactSheet + CallButton + matrice de rôles.
  • ReferralDetailScreen._onContactProprio() déjà câblé sur proprio.telephone.

Le delta réel à traiter

  • Parsing Appartement : les 2 champs aplatis communeNom et villeNom du nouveau DTO n'étaient pas exploités.
  • Règle R4 : les réservations MANUELLE avec client externe du proprio affichaient toujours les infos du client côté démarcheur.

3. Changements implémentés

Trois fichiers modifiés, environ 25 lignes utiles (hors docstrings). Aucun nouveau fichier créé.

3.1 lib/model/residence/appart.dart

  • Ajout de 2 champs nullables : String? communeNom et String? villeNom.
  • Parsing dans fromJson, sérialisation dans toJson, intégration au constructeur et au copyWith.
  • Nouveau getter localiteLabel qui retourne "Commune, Ville" en priorité (champs aplatis backend), avec fallback sur l'Address nested legacy via address.locationDisplayName.
String get localiteLabel {
  if (communeNom != null && villeNom != null) {
    return '$communeNom, $villeNom';
  }
  if (communeNom != null) return communeNom!;
  if (villeNom != null) return villeNom!;
  if (address?.hasFallbackLocation == true) {
    return address!.locationDisplayName;
  }
  return '';
}

3.2 lib/screen/client/demarcheur/referrals/widget/referral_display.dart

  • Import ajouté : package:asfar/model/enumeration/reservation_type.dart.
  • Nouveau getter isClientConfidential (source unique de la règle R4).
  • Adaptation de referralClientName et referralClientPhone qui court-circuitent désormais sur l'état confidentiel.
bool get isClientConfidential => type == ReservationType.manuelle;

String get referralClientName {
  if (isClientConfidential) return 'Client confidentiel';
  final base = clientNom?.trim().isNotEmpty == true
      ? clientNom!
      : 'Client #${id ?? 0}';
  return base;
}

String get referralClientPhone {
  if (isClientConfidential) return '';
  return clientExterneTelephone ?? locataire?.telephone ?? '';
}

3.3 lib/screen/client/demarcheur/referrals/referral_detail_screen.dart

  • Section « Client » conditionnée par un spread if (!reservation.isClientConfidential) ...[].
  • Aucune modification d'imports — l'extension ReferralDisplay apporte le nouveau getter.

4. Règle métier R4 — Anonymisation client externe

Énoncé : Quand une réservation est de type MANUELLE et que le client est un externe passé par le proprio (clientExterneNom renseigné, locataire == null), le démarcheur ne voit PAS les infos du client : seuls l'appartement, les dates, le statut et la commission sont visibles.

Décision UI validée — Option A (masquage complet)

Plutôt qu'afficher une card neutre « Client confidentiel » (option B) ou un bandeau discret (option C), l'option retenue masque entièrement la section « Client » sur l'écran détail référence. Le scroll passe directement de Logement à Propriétaire.

Cas couverts

Type de réservation Locataire Client externe Affichage section Client
DEMARCHEUR null renseigné ✓ visible (le démarcheur réfère son propre client externe)
PLATEFORME renseigné null ✓ visible (locataire de la plateforme)
MANUELLE null renseigné (du proprio) ✗ masquée (R4 — confidentialité)

5. Hors-périmètre (explicite)

  • R5 distinction « ma demande » vs « autre démarcheur » : non implémentée. L'endpoint backend ne renvoie que les réservations du démarcheur connecté (validé en BA). À reconsidérer si l'API évolue pour exposer les demandes d'autres démarcheurs sur les mêmes appartements partenaires.
  • Nouveau widget de contact : aucun. Le système existant (ContactTargetResolver, ContactSheet, CallButton) couvre déjà le besoin et a été simplement vérifié.
  • Modèle Reservation : pas touché — parsing déjà aligné.
  • BLoC et Service démarcheur : inchangés — fonctionnels.
  • Tests unitaires : non ajoutés (la feature ne les exigeait pas), suggérés par l'audit en amélioration de suite (cf. § Qualité).
