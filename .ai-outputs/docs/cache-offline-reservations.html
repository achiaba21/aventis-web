Cache Offline Reservations

Systeme de cache local Hive pour les reservations avec pattern cache-first

Resume

Cette feature permet aux proprietaires de consulter leurs reservations meme sans connexion au serveur grace au stockage local Hive.

  • Cache-first - Affiche les donnees du cache immediatement, puis rafraichit en arriere-plan
  • Mode offline - Fonctionne sans serveur si des donnees ont deja ete mises en cache
  • Fallback gracieux - Retourne une liste vide au lieu d'une erreur si offline et cache vide
  • Synchronisation transparente - Met a jour l'UI automatiquement quand les donnees API arrivent

Probleme Resolu

Avant

Sans serveur, l'interface proprietaire affichait "Erreur de recuperation des reservations" car le BLoC appelait directement l'API sans fallback.

Apres

Le nouveau ReservationRepository utilise le pattern cache-first :

  • Serveur ON + cache vide → Charge depuis API, sauvegarde en cache
  • Serveur OFF + cache vide → Liste vide (pas d'erreur)
  • Serveur ON + cache rempli → Affiche cache, rafraichit en arriere-plan
  • Serveur OFF + cache rempli → Affiche cache

Architecture

Pattern Cache-First

UI → ReservationBloc → ReservationRepository
↓
StorageService (Hive) ←→ ReservationService (API)

Composants

ReservationRepository

Singleton qui orchestre le cache-first. Retourne immediatement les donnees du cache et lance un rafraichissement en arriere-plan.

ReservationResult

Wrapper contenant la liste des reservations et un boolean isFromCache indiquant la source des donnees.

StorageService

Service de stockage Hive centralise. Nouvelle box reservationsBox ajoutee pour les reservations.

UpdateReservationsFromApi

Nouvel event BLoC emis quand les donnees API arrivent en arriere-plan pour mettre a jour l'UI.

Diagramme de Sequence

1. UI declenche LoadProprietaireReservations
2. ReservationBloc appelle Repository.getProprietaireReservations()
3. Repository verifie le cache Hive
4. Si cache non vide:
   a. Retourne ReservationResult(cached, isFromCache: true)
   b. Lance _refreshInBackground() avec callback onApiData
5. BLoC emet ReservationLoaded avec donnees cache
6. En arriere-plan: API appele, cache mis a jour
7. Callback onApiData declenche add(UpdateReservationsFromApi)
8. BLoC emet nouveau ReservationLoaded avec donnees fraiches

API du Repository

Methode Parametres Retour Description
getProprietaireReservations forceRefresh, onApiData Future<ReservationResult> Reservations du proprietaire (cache-first)
getUserReservations forceRefresh, onApiData Future<ReservationResult> Reservations de l'utilisateur (cache-first)
getCachedReservations - List<Reservation> Lecture directe du cache
fetchAndCacheProprietaireReservations - Future<List<Reservation>> Force appel API + sauvegarde cache
clearCache - Future<void> Vide le cache des reservations
isCacheStale maxAgeHours bool Verifie si le cache est perime

Utilisation

Chargement standard (cache-first)

// Dans le BLoC - deja implemente
on<LoadProprietaireReservations>((event, emit) async {
  emit(ReservationLoading());

  final result = await _repository.getProprietaireReservations(
    forceRefresh: false,
    onApiData: (apiReservations) {
      // Mise a jour quand API repond
      add(UpdateReservationsFromApi(apiReservations));
    },
  );

  emit(ReservationLoaded(result.reservations));
});

Force refresh

// Pour forcer le rafraichissement depuis l'API
final result = await _repository.getProprietaireReservations(
  forceRefresh: true,
);
// En cas d'erreur API, retourne quand meme le cache

Verification du cache

// Verifier si les donnees viennent du cache
if (result.isFromCache) {
  // Afficher un indicateur "donnees locales"
}

// Verifier si le cache est perime (> 1 heure)
if (_repository.isCacheStale(maxAgeHours: 1)) {
  // Forcer un refresh
}

StorageService - Nouvelles Methodes

Methode Description
